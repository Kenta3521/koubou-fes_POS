# 光芒祭POSシステム要件定義書

**作成日:** 2026年2月2日
**作成者:** 岩田 宗也
**版数:** 1.0

## 1. プロジェクト概要

### 1.1 背景と目的

本学の光芒祭において、各出店団体（クラス・サークル）の会計業務を効率化し、実行委員会による売上管理の透明性と正確性を確保する。従来の紙ベースまたは電卓ベースの集計におけるミスや不正を防ぎ、現金およびPayPay決済を一元管理できるシステムを構築する。

### 1.2 スコープ

* **対象ユーザー:** 出店団体のスタッフ・責任者、および実行委員会（システム管理者）。
* **対象業務:** 注文入力、会計処理（現金/PayPay）、売上集計、在庫管理。
* **利用環境:**
    * **POS端末:** スマートフォン（Webアプリ）がメイン。タブレット/PCにも対応。
    * **管理画面:** タブレット/PCブラウザ推奨。



---

## 2. システム構成

### 2.1 アーキテクチャ

「サーバー集中管理型」を採用し、データの整合性を担保する。

* **バックエンド:** Node.js (Express) + Prisma + Socket.io
* データのCRUD、PayPay APIとの通信、Webhook処理、リアルタイム配信を担当。


* **フロントエンド:** React + TypeScript + Vite (PWA対応)
* UI/UXの提供。PWAとしてスマートフォンのホーム画面に追加可能。


* **データベース:** SQLite (開発時) / PostgreSQL (本番推奨)
* 実行委員会が管理する単一のDBサーバーを利用。


* **決済連携:** PayPay Open Payment API (User Scan方式)
* 外部公開URL (ngrok等) を経由してWebhookを受信。



---

## 3. ユーザー種別と権限（ロール）

| ロール名 | 対象者 | 役割と権限 |
| --- | --- | --- |
| **SYSTEM ADMIN** | 実行委員会 | **全権限**。全団体のデータ閲覧・編集、強制返金、システム緊急停止、アカウント管理。 |
| **ORG ADMIN** | 団体リーダー | **自団体のみ**。メニュー編集、割引設定、スタッフ管理、自団体の売上分析、返金処理。 |
| **STAFF** | 一般シフト | **自団体のみ**。レジ打ち（販売）、在庫切れ切替。（返金・編集は不可） |

## 4. 機能要件

### 4.1 共通機能

* **認証機能:**
    * メールアドレス/パスワードによるログイン。
    * JWTを用いたセッション管理。
    * **管理者登録フロー:** 登録直後は全員 `STAFF` となる。責任者は登録後、実行委員会（SYSTEM ADMIN）へ申請し、管理画面より `ORG_ADMIN` ロールへの変更を受けることで管理者権限を取得する。


* **団体切替:**
* 複数団体に所属するユーザーが、操作対象の団体を選択・切替できる機能。



### 4.2 POSレジ機能 (STAFF / ORG ADMIN)

* **注文入力:**
* 商品ボタン（タッチパネル対応UI）によるカート追加。
* 数量変更、カート内削除。


* **割引適用:**
* 事前登録された「割引プリセット（例: タイムセール）」の選択適用。
* *制約: スタッフによる自由な金額入力による値引きは禁止。*


* **会計処理:**
* **現金:** 預かり金入力、お釣り計算、完了処理。
* **PayPay:** 動的QRコード生成、決済完了の自動検知（ポーリング不要）、完了画面への自動遷移。
* **PayPayタイムアウト:** 5分経過で自動キャンセル、残り時間のカウントダウン表示、手動キャンセルボタン。



### 4.3 ダッシュボード・管理機能 (ORG ADMIN)

* **カテゴリ管理:** 商品カテゴリの追加・編集・削除（例: フード、ドリンク、グッズ）。
* **メニュー管理:** 商品の追加・編集・削除、カテゴリ割り当て。
* **在庫管理:** 「売り切れ（Sold Out）」ステータスの手動切り替え。在庫数量の補充・調整機能。変動履歴の記録。
* **売上分析:** リアルタイム売上グラフ、商品別販売数ランキングの表示。
* **取引履歴:** 過去の取引ログ確認と、PayPay全額返金処理（API連動）。

### 4.4 システム管理機能 (SYSTEM ADMIN)

* **全体モニタリング:** 全店舗の稼働状況、総売上のリアルタイム監視、カテゴリ別売上動向。
* **テナント管理:** 出店団体の新規作成、招待コード発行。
* **監査ログ:** 不正防止のため、全ての決済・修正・返金操作のログを閲覧可能にする。

## 5. データ要件（データベース設計）

### 5.1 ER図概要

多対多のリレーションを用い、柔軟な所属管理を実現する。

* **Organization:** 団体情報。
* **User:** 個人アカウント。`isSystemAdmin` フラグを持つ。
* **UserOrganization:** 中間テーブル。ロール(`ADMIN`/`STAFF`)を保持。
* **Category:** 団体ごとの商品カテゴリ（例: フード、ドリンク）。
* **Product:** 団体ごとの商品。`categoryId` でカテゴリに紐づく。
* **Discount:** 団体ごとの割引設定。
* **Transaction:** 会計ヘッダー。`paypayOrderId` で決済を特定。
* **TransactionItem:** 会計明細。販売時点の単価(`unitPrice`)を固定記録。

## 6. 非機能要件

### 6.1 可用性・耐障害性

* **ネットワーク耐性:**
* Wi-Fi切断時でも、現金決済のみ継続可能とする（ローカルストレージへの一時保存と復帰後の同期）。
* PayPay決済時は、通信切断時に即座にエラーを表示し、現金への誘導を行う。


* **高負荷対策:**
* Socket.ioのイベントを軽量化し、全台同時アクセス時のサーバーダウンを防ぐ。



### 6.2 セキュリティ

* **なりすまし防止:**
* PayPayの決済完了通知は、Webhookの署名検証を必須とする。
* クライアント（フロントエンド）からの「決済完了通知」は一切信用しない。


* **データ分離:**
* APIレベルで `organizationId` を検証し、他団体のデータへのアクセスを遮断する。



### 6.3 UX/UI

* **shadcn/ui:** コンポーネントライブラリを使用し、統一感のあるモダンなデザインとする。
* **ダークモード:** 屋外テントなどの環境に合わせて視認性を確保する。
