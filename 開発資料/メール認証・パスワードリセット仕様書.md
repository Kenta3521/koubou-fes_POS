# メール認証・パスワードリセット仕様書

**作成日:** 2026年3月19日
**ステータス:** ドラフト
**目的:** メール送信基盤（AWS SES）の導入、新規アカウント作成フローへのメール認証追加、およびパスワードリセットフローの本番化を行う。

---

## 1. 概要

### 1.1 背景

現行システムでは以下の課題がある。

- **新規アカウント作成:** メールアドレスの所有確認を行わずにアカウントが即時作成される
- **パスワードリセット:** リセットトークンをAPIレスポンスに含める開発用措置のままであり、メール送信が未実装
- **メール送信基盤:** 今後の電子レシート発行等を見据え、汎用的なメール送信基盤が必要

### 1.2 スコープ

| 対象 | 内容 |
|------|------|
| メール送信基盤 | AWS SES を利用した送信サービスの実装 |
| アカウント作成フロー | メールアドレス検証を含む2ステップ登録への変更 |
| パスワードリセットフロー | メールベースの標準的なリセットフローへの移行 |
| 対象クライアント | Web管理画面のみ（iOSアプリは対象外） |

### 1.3 前提条件

- AWS SES は本番環境（サンドボックス解除済み）
- 送信ドメイン: `ukeja.com`（SES で検証済み）
- 既存ユーザーのメールアドレスは検証済みとして扱う

## 2. メール送信基盤（AWS SES）

### 2.1 技術構成

| 項目 | 値 |
|------|-----|
| サービス | Amazon Simple Email Service (SES) |
| SDK | `@aws-sdk/client-ses` (v3) |
| 送信元アドレス | `noreply@ukeja.com` |
| リージョン | 環境変数で指定 |

### 2.2 環境変数

| 変数名 | 必須 | 説明 | 例 |
|--------|------|------|----|
| `AWS_REGION` | Yes | SES リージョン | `ap-northeast-1` |
| `AWS_ACCESS_KEY_ID` | Yes | IAM アクセスキー | — |
| `AWS_SECRET_ACCESS_KEY` | Yes | IAM シークレットキー | — |
| `SES_FROM_ADDRESS` | Yes | 送信元アドレス | `noreply@ukeja.com` |
| `APP_BASE_URL` | Yes | フロントエンドのベースURL | `https://pos.ukeja.com` |

## 3. 新規アカウント作成フロー

### 3.1 現行フロー（廃止）

```
ユーザー → メール + パスワード + 名前 + 招待コード を入力
       → POST /auth/register
       → アカウント即時作成 & JWT返却
```

### 3.2 新フロー（2ステップ登録）

```
Step 1: メールアドレス + 招待コード入力
        ↓ POST /auth/register/request
        サーバーが招待コード検証 → 検証トークン付きメール送信
        ↓
Step 2: メール内リンクをクリック
        → GET /auth/register/verify/:token（トークン検証 → フロントへリダイレクト）
        → フロントで名前 + パスワード入力画面を表示
        ↓ POST /auth/register/complete
        アカウント作成完了 & JWT返却
```

### 3.3 DBスキーマ変更

#### EmailVerification モデル（新規追加）

```prisma
model EmailVerification {
  id         String   @id @default(uuid())
  email      String
  inviteCode String
  token      String   @unique
  expiresAt  DateTime
  used       Boolean  @default(false)
  createdAt  DateTime @default(now())

  @@index([email])
  @@index([token])
}
```

| フィールド | 型 | 説明 |
|-----------|------|------|
| `id` | UUID | 主キー |
| `email` | String | 登録申請メールアドレス |
| `inviteCode` | String | 申請時に入力された招待コード |
| `token` | String (unique) | 検証トークン（64文字hex） |
| `expiresAt` | DateTime | 有効期限 |
| `used` | Boolean | 使用済みフラグ |
| `createdAt` | DateTime | 作成日時 |

#### User モデルへのフィールド追加

```prisma
model User {
  // ... 既存フィールド
  emailVerified Boolean @default(false)  // 追加
}
```

既存ユーザーへの対応として、マイグレーション時に既存レコードの `emailVerified` を `true` に設定する。

### 3.4 APIエンドポイント

#### 3.4.1 `POST /auth/register/request` — 登録申請

メールアドレスと招待コードを受け取り、検証メールを送信する。

**Request:**
```json
{
  "email": "newuser@example.com",
  "inviteCode": "ABC123"
}
```

**バリデーション:**
- `email`: 必須、メールアドレス形式
- `inviteCode`: 必須、1文字以上

**処理フロー:**

1. 招待コードで組織を検索し、有効性を検証
2. メールアドレスが既存ユーザーと重複していないか確認
3. 同一メールアドレスの未使用 `EmailVerification` を無効化（`used: true`）
4. 検証トークン生成: `crypto.randomBytes(32).toString('hex')`（64文字hex）
5. `EmailVerification` レコード作成（有効期限: 24時間）
6. 検証メール送信

**Response (200):**
```json
{
  "success": true,
  "data": {
    "message": "確認メールを送信しました"
  }
}
```

> **セキュリティ:** メールアドレスの存在有無に関わらず同一レスポンスを返す。招待コードが無効な場合のみエラーを返す（招待コードは秘密情報ではないため）。

**エラーレスポンス:**

| ステータス | code | 条件 |
|-----------|------|------|
| 400 | `VALIDATION_ERROR` | バリデーションエラー |
| 400 | `ORG_INVALID_INVITE_CODE` | 招待コードが無効 |
| 400 | `ORG_INACTIVE` | 組織が無効化されている |

#### 3.4.2 `GET /auth/register/verify/:token` — トークン検証

メール内リンクのクリック時に呼ばれる。トークンを検証し、フロントエンドの登録完了画面にリダイレクトする。

**処理フロー:**

1. `token` で `EmailVerification` を検索
2. 存在・未使用・期限内であることを確認
3. 有効な場合: `{APP_BASE_URL}/register/complete?token={token}&email={email}` にリダイレクト (302)
4. 無効な場合: `{APP_BASE_URL}/register/expired` にリダイレクト (302)

#### 3.4.3 `POST /auth/register/complete` — 登録完了

検証済みトークンとユーザー情報を受け取り、アカウントを作成する。

**Request:**
```json
{
  "token": "abc123...",
  "name": "田中太郎",
  "password": "securePassword123"
}
```

**バリデーション:**
- `token`: 必須
- `name`: 必須、1文字以上
- `password`: 必須、8文字以上

**処理フロー:**

1. `token` で `EmailVerification` を検索・検証（存在・未使用・期限内）
2. `EmailVerification` から `email` と `inviteCode` を取得
3. 招待コードで組織を再検証（申請後に組織が無効化された場合への対応）
4. メールアドレスの重複チェック（申請後に他者が同じメールで登録した場合への対応）
5. トランザクション内で以下を実行:
   - ユーザー作成（`emailVerified: true`）
   - 組織メンバーシップ作成
   - デフォルトロール（スタッフ）割り当て
   - `EmailVerification` を `used: true` に更新
6. JWT発行
7. 監査ログ記録（`USER_REGISTER`）

**Response (201):**
```json
{
  "success": true,
  "data": {
    "token": "eyJhbGciOiJIUzI1NiIs...",
    "user": {
      "id": "uuid",
      "email": "newuser@example.com",
      "name": "田中太郎",
      "status": "ACTIVE",
      "isSystemAdmin": false,
      "createdAt": "2026-03-19T00:00:00.000Z",
      "organizations": [
        {
          "id": "org-uuid",
          "name": "団体名",
          "isActive": true,
          "role": "スタッフ",
          "permissions": ["view_product", "create_transaction", ...]
        }
      ]
    }
  }
}
```

**エラーレスポンス:**

| ステータス | code | 条件 |
|-----------|------|------|
| 400 | `VALIDATION_ERROR` | バリデーションエラー |
| 400 | `INVALID_OR_EXPIRED_TOKEN` | トークンが無効・期限切れ・使用済み |
| 400 | `ORG_INVALID_INVITE_CODE` | 組織が無効化された |
| 409 | `EMAIL_ALREADY_EXISTS` | メールアドレスが既に使用されている |

#### 3.4.4 `POST /auth/register`（既存） — 廃止

新フロー移行後に廃止する。移行期間中は両方のエンドポイントを共存させてもよいが、最終的には削除する。

## 4. パスワードリセットフロー

### 4.1 現行フロー（廃止）

```
ユーザー → POST /auth/reset-password-request (email)
        ← リセットトークンがレスポンスに含まれる（開発用措置）
        → POST /auth/reset-password (token, newPassword)
```

### 4.2 新フロー

```
Step 1: メールアドレス入力
        ↓ POST /auth/reset-password-request
        サーバーがリセットメール送信
        ↓
Step 2: メール内リンクをクリック
        → フロントで新パスワード入力画面を表示
        ↓ POST /auth/reset-password
        パスワード更新完了
```

### 4.3 DBスキーマ変更

#### PasswordReset モデル（新規追加）

```prisma
model PasswordReset {
  id        String   @id @default(uuid())
  userId    String
  token     String   @unique
  expiresAt DateTime
  used      Boolean  @default(false)
  createdAt DateTime @default(now())

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId])
  @@index([token])
}
```

| フィールド | 型 | 説明 |
|-----------|------|------|
| `id` | UUID | 主キー |
| `userId` | String (FK) | リセット対象ユーザー |
| `token` | String (unique) | リセットトークン（64文字hex） |
| `expiresAt` | DateTime | 有効期限 |
| `used` | Boolean | 使用済みフラグ |
| `createdAt` | DateTime | 作成日時 |

> **注:** 現行のJWTベースのリセットトークン（`generateResetToken` / `verifyResetToken`）はDBベースのトークンに置き換える。JWTリセットトークンは一度発行すると無効化できないが、DBベースなら明示的に無効化できる。

### 4.4 APIエンドポイント（既存改修）

#### 4.4.1 `POST /auth/reset-password-request` — リセット要求（改修）

**Request:**
```json
{
  "email": "user@example.com"
}
```

**処理フロー:**

1. メールアドレスでユーザーを検索
2. ユーザーが存在し、かつ ACTIVE または SUSPENDED の場合:
   - 同一ユーザーの未使用 `PasswordReset` を無効化（`used: true`）
   - トークン生成: `crypto.randomBytes(32).toString('hex')`
   - `PasswordReset` レコード作成（有効期限: **15分**）
   - リセットメール送信
3. ユーザーが存在しない場合もエラーを返さない

**Response (200):**
```json
{
  "success": true,
  "data": {
    "message": "登録されているメールアドレスであれば、リセット用のメールを送信しました"
  }
}
```

> **セキュリティ:** ユーザーの存在有無に関わらず同一レスポンス・同一レスポンスタイムを返す。

**変更点（現行との差分）:**

| 項目 | 現行 | 変更後 |
|------|------|--------|
| トークン生成 | JWT (`generateResetToken`) | `crypto.randomBytes(32)` + DB保存 |
| トークン配信 | APIレスポンスに含む | メール送信 |
| ユーザー未存在時 | エラーを返す | 成功レスポンスを返す |
| トークン無効化 | 不可（JWT有効期限のみ） | DBレコードの `used` フラグ |

#### 4.4.2 `POST /auth/reset-password` — リセット実行（改修）

**Request:**
```json
{
  "token": "abc123...",
  "newPassword": "newSecurePassword123"
}
```

**処理フロー:**

1. `token` で `PasswordReset` を検索
2. 存在・未使用・期限内であることを確認
3. パスワードをbcryptでハッシュ化
4. トランザクション内で以下を実行:
   - ユーザーのパスワード更新
   - `PasswordReset` を `used: true` に更新
5. 監査ログ記録（`PASSWORD_RESET`）

**Response (200):**
```json
{
  "success": true,
  "data": {
    "message": "パスワードを再設定しました"
  }
}
```

**エラーレスポンス:**

| ステータス | code | 条件 |
|-----------|------|------|
| 400 | `VALIDATION_ERROR` | バリデーションエラー |
| 400 | `INVALID_OR_EXPIRED_TOKEN` | トークンが無効・期限切れ・使用済み |

**変更点（現行との差分）:**

| 項目 | 現行 | 変更後 |
|------|------|--------|
| トークン検証 | JWT検証 (`verifyResetToken`) | DB検索 + 状態チェック |
| トークン無効化 | なし | `used: true` に更新 |
| 監査ログ | なし | `PASSWORD_RESET` を記録 |

## 5. セキュリティ

### 5.1 トークン仕様

| 項目 | アカウント登録 | パスワードリセット |
|------|--------------|------------------|
| 生成方法 | `crypto.randomBytes(32).toString('hex')` | 同左 |
| 長さ | 64文字 (hex) | 同左 |
| エントロピー | 256bit | 同左 |
| 有効期限 | 24時間 | 15分 |
| 使用回数 | 1回限り | 1回限り |
| 保存先 | DB (`EmailVerification`) | DB (`PasswordReset`) |

### 5.2 メール列挙攻撃対策

- 登録申請・パスワードリセットの両エンドポイントは、メールアドレスの存在有無に関わらず**同一レスポンス**を返す
- レスポンスタイムの差異によるサイドチャネルを防ぐため、ユーザー未存在時も一定の処理を行う

### 5.3 レートリミット

メール送信を伴うエンドポイントには追加のレートリミットを設ける。

| エンドポイント | 制限 |
|--------------|------|
| `POST /auth/register/request` | 同一メールアドレスにつき1分間に1回 |
| `POST /auth/reset-password-request` | 同一メールアドレスにつき1分間に1回 |
| `POST /auth/register/complete` | 同一トークンにつき1分間に5回（総当たり対策） |
| `POST /auth/reset-password` | 同一IPにつき1分間に5回 |

### 5.4 トークンの無効化

- **再発行時:** 同一メール/ユーザーの未使用トークンは自動的に無効化される（有効なトークンは常に1つのみ）
- **使用時:** `used: true` に更新され再利用不可
- **期限切れ:** `expiresAt` を超過したトークンは検証時に拒否される
- **定期クリーンアップ:** 期限切れトークンの定期削除は任意（パフォーマンス要件に応じて後日検討）

### マイグレーション手順

1. `emailVerified` カラムを `User` に追加（デフォルト: `false`）
2. 既存ユーザーの `emailVerified` を `true` に一括更新
3. `EmailVerification` テーブル作成
4. `PasswordReset` テーブル作成

```sql
-- マイグレーション参考SQL
```

## 7. 監査ログ

以下のアクションを監査ログに記録する。

| アクション | カテゴリ | 記録タイミング | payload |
|-----------|---------|--------------|---------|
| `REGISTER_REQUEST` | AUTH | 登録申請時 | `{ email }` |
| `USER_REGISTER` | AUTH | 登録完了時（既存） | `{ userId, email, organizationId }` |
| `PASSWORD_RESET_REQUEST` | AUTH | リセット要求時 | `{ email }` |
| `PASSWORD_RESET` | AUTH | リセット完了時 | `{ userId }` |

> **注:** `REGISTER_REQUEST` と `PASSWORD_RESET_REQUEST` は `userId` が確定しない段階のため、システムユーザーまたは匿名として記録する設計を検討する。
