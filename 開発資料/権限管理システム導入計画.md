# 権限管理システム（RBAC）導入計画書

## 1. 概要
現在のシステムでは、権限管理が「コードベース（Enumと条件分岐）」で実装されています。より柔軟な権限設定（例：カスタムロールの作成、権限の動的変更）を可能にするため、データベース駆動型の **Role-Based Access Control (RBAC)** への移行を計画します。

本ドキュメントでは、AD（Active Directory）ライクな一括管理を実現するためのデータ構造と、**CASL (Common Ability Schema Language)** を採用した実装ステップを定義します。

**CASL採用のメリット:**
1.  **Frontend/Backendのロジック共有**: 同じ権限ルールを両側で利用可能。
2.  **属性ベース権限 (ABAC)**: 「自分の所属する団体の商品のみ編集可能」といった条件付き権限を記述しやすい。
3.  **標準化**: 自己流の `if (role === 'admin')` の乱立を防ぐ。

## 2. データベース設計変更 (Schema Changes)

現在の `Enum Role` を廃止し、以下のテーブル構成へ移行します。

### 新規追加・変更テーブル

#### 1. Permissions (権限マスター)
システム内で実行可能な「操作」を定義します。
```prisma
model Permission {
  id          String   @id @default(uuid())
  code        String   @unique // 例: "dashboard:view", "product:create"
  description String?
  category    String   // 例: "System", "Inventory", "Sales"
  createdAt   DateTime @default(now())
  roles       RolePermission[]
}
```

#### 2. ServiceRoes (旧 Role Enum の代替テーブル)
組織内で割り当てる「役割（ロール）」を定義します。
※ `Role` という名前は既存Enumと競合する可能性があるため、移行中は注意が必要です。
```prisma
model ServiceRole {
  id             String   @id @default(uuid())
  organizationId String?  // nullの場合はシステム共通ロール（SystemAdmin等）
  name           String   // 例: "店長", "アルバイト", "システム管理者"
  isSystemRole   Boolean  @default(false) // 削除不可のシステム定義ロール
  createdAt      DateTime @default(now())
  
  permissions    RolePermission[]
  users          UserOrganization[] // 既存のリレーション先を変更
  organization   Organization? @relation(fields: [organizationId], references: [id])
}
```

#### 3. RolePermission (中間テーブル)
ロールと権限の多対多関係を管理します。
```prisma
model RolePermission {
  roleId       String
  permissionId String
  role         ServiceRole @relation(fields: [roleId], references: [id])
  permission   Permission  @relation(fields: [permissionId], references: [id])

  @@id([roleId, permissionId])
}
```

#### 4. UserOrganization (変更)
既存の `role Role @default(STAFF)` を外部キー参照に変更します。
```prisma
model UserOrganization {
  // ...既存フィールド
  // role Role @default(STAFF)  <-- 削除
  roleId String
  role   ServiceRole @relation(fields: [roleId], references: [id])
}
```

## 3. バックエンド実装 (API Layer)

### ミドルウェアの刷新 (CASL Integration)

CASLの `Ability` インスタンスを生成し、それを用いてチェックを行います。

**1. Ability Factory (権限生成ロジック):**
```typescript
import { AbilityBuilder, createMongoAbility } from '@casl/ability';

export function defineAbilityFor(user: UserWithPermissions) {
  const { can, cannot, build } = new AbilityBuilder(createMongoAbility);

  // DBから取得した権限を展開
  user.permissions.forEach(p => {
    // p.action = 'manage', p.subject = 'Product' など
    can(p.action, p.subject); 
  });

  // 条件付き権限の例 (ABAC)
  // "自分が所属する組織のデータのみ"
  can('manage', 'Organization', { id: user.organizationId });

  return build();
}
```

**2. ミドルウェア:**
```typescript
// 特定の権限を持っているかチェック
router.get(
  '/products/:id',
  (req, res, next) => {
    const ability = defineAbilityFor(req.user);
    // "Product" リソースの "read" アクションが可能か？
    if (ability.can('read', 'Product')) {
      return next();
    }
    return res.status(403).json({ error: 'Forbidden' });
  }
);
```

### APIエンドポイントの追加
権限管理画面用のAPIを追加します。
- `GET /api/v1/permissions`: 全権限リスト取得
- `GET /api/v1/roles`: ロール一覧取得
- `POST /api/v1/roles`: カスタムロール作成
- `PUT /api/v1/roles/:id`: ロールの権限設定変更

## 4. フロントエンド実装 (Web Layer)

### CASL React Hooks の導入
`@casl/react` を導入し、`<Can>` コンポーネントやフックを利用します。

```typescript
import { useAbility } from '@casl/react';
import { AbilityContext } from './Can';

// コンポーネント内での使用
const ability = useAbility(AbilityContext);

if (ability.can('delete', 'Product')) {
  return <DeleteButton />;
}
```

### ガードコンポーネントの作成
ルート保護もCASLで行います。

```typescript
<RequirePermission action="read" subject="Dashboard">
  <DashboardPage />
</RequirePermission>
```

### 管理画面の構築
- **ロール管理画面**: ロールの作成・編集、権限のチェックボックス割り当て画面。
- **権限マトリクス**: どのロールが何ができるかを一覧表示する画面。

## 5. 導入ステップ (Migration Plan)

### Phase A: 基盤構築 (並行稼働)
1.  `Permissions`, `ServiceRole`, `RolePermission` テーブルを作成。
2.  初期データ（Seed）を作成し、既存の `ADMIN`, `STAFF` に相当するロールと権限を投入。
3.  `UserOrganization` に `roleId` カラムを追加し（nullable）、既存データの `role` Enumの値に基づいて `roleId` を埋めるデータ移行スクリプトを実行。

### Phase B: バックエンド切替
1.  `auth.ts` ミドルウェアを更新し、権限チェックロジックを実装。
2.  各コントローラーのルート定義に `requirePermission` を適用。
3.  テストを実行し、既存のアクセス制御が壊れていないか確認。

### Phase C: フロントエンド切替 & 管理機能解放
1.  フロントエンドの `useAuth` やストアを更新し、ログイン時に `permissions` 配列を受け取るように修正。
2.  `isSystemAdmin` などのハードコードを `can('system:manage')` などに置き換え。
3.  「ロール管理画面」を実装し、ユーザーに権限設定機能を開放。

### Phase D: 詳細な権限設定とクリーンアップ

ユーザーの要望に基づき、**すべてのAPI・画面・団体へのアクセス**を個別に設定可能にします。これにより、システム管理者や団体管理者がよりきめ細かく権限を制御できるようにします。

#### 1. 権限の細分化 (Audit & Expansion)
`API設計書.md` に基づき、すべてのリソース操作を網羅する権限コードを追加・定義します。

| :--- | :--- | :--- |

#### 2. 実装ステップ

**Step 1: Seed & Database更新**
- `apps/api/src/scripts/seed_rbac.ts` を更新し、上記の全権限を登録する。
- デフォルトの `ADMIN`, `STAFF` ロール定義を更新し、新しい権限セットを含める。

**Step 2: バックエンド制御 (API)**
- `apps/api/src/controllers/` を網羅的に確認。
- すべての保護されたルートに対して `requirePermission('<code')` ミドルウェアを適用する。
- `isSystemAdmin` フラグへの依存を減らし、可能な限り `system:manage` 権限チェックへ移行する。

**Step 3: フロントエンド制御 (UI)**
- 各ページ・コンポーネントで `<Can I="..." a="..." />` や `useAbility()` を使用し、権限のないボタン（編集・削除・返金など）を非表示/無効化する。
- 権限マトリクス編集画面 (`RoleEditPage`) が新しい全権限を表示・保存できることを確認。

**Step 4: Cleanup (Legacy Removal)**
- `UserOrganization` テーブルから旧 `role` Enum カラムを削除。
- コードベースから `Role` Enum への依存を完全に排除。
