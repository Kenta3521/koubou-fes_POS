# iOS ネイティブ POS アプリ 仕様書

**作成日:** 2026年3月14日
**版数:** 1.0

---

## 1. 目的と位置づけ

本システムは、既存の「光芒祭POSシステム（Web版）」のうち、**店頭でのレジ操作（POS機能）** に特化した iOS ネイティブアプリである。
Web版の主な課題である「より直感的な操作感」「NFCによるタッチ決済（Tap to Pay on iPhone）の実現」を目的として開発される。

### 1.1 システムのスコープ

*   **対象ユーザー**: 各出店団体のスタッフ（STAFF）および管理者（ORG_ADMIN）。
*   **対象機能**: ログイン、団体選択、注文入力、カート管理、支払処理（現金・PayPay・**Tap to Pay**）、直近の注文履歴のみ。
*   **対象外機能**: カテゴリ管理、商品管理、割引設定、スタッフ管理、売上サマリーなどの「管理機能」は全て Web 版で行う前提とする。

---

## 2. アプリ構成とデザイン方針

iOSの標準的なヒューマンインターフェイスガイドライン（HIG）に沿い、スマートフォン（iPhone）での迅速なレジ操作に最適化する。

### 2.1 ナビゲーション（TabView）

画面下部にタブバーを配置し、主要な機能へ1タップでアクセス可能にする。

1.  **🛒 レジ (POS)**: メインの注文入力画面。
2.  **📜 注文履歴**: その日に販売した直近の取引一覧（トラブル時の確認・操作ミスによるキャンセル・返金用）。
3.  **⚙️ 設定**: 現在のログインユーザー情報、所属団体名、ログアウト操作。

### 2.2 UIのiOS最適化

*   **ハーフモーダル (Sheet)**:
    *   **カート**: 注文入力画面では「カートを見る」ボタンを画面下部に固定配置し、タップ時にハーフモーダルでカート内容を引き上げるスタイルとする（商品一覧画面を広く確保するため）。
    *   **割引選択**: 注文確認画面からの割引プリセット選択もハーフモーダルで表示する。
*   **Swipe Actions**:
    *   カート内の商品削除は、リストの行を左スワイプして削除（Delete）ボタンを出すネイティブな操作を採用。
*   **Haptics (触覚フィードバック)**:
    *   「商品をカートに追加した時」「決済が完了した時」「エラー発生時」に短い振動を追加し、喧騒の多い環境（学祭）でも操作感が伝わるようにする。
*   **ダークモード**:
    *   システム設定に連動、またはアプリ内設定で固定可能とする（屋外テント等での視認性向上のため）。

## 3. 画面遷移と機能詳細

### 3.1 認証・初期設定

| 画面名 | 概要・仕様 |
| :--- | :--- |
| **ログイン画面** | メールアドレスとパスワードによる認証。<br>認証成功後、JWTトークンを**Keychain**に安全に保存し、次回起動時は生体認証（Face ID等）または自動でログインを再開する。 |
| **団体選択画面** | 所属団体が複数ある場合に表示。選択した団体IDを設定し、レジ画面へ遷移する。1つしか所属していない場合はスキップ。 |

### 3.2 🛒 レジタブ (POSフロー)

| 画面名 | 概要・仕様 |
| :--- | :--- |
| **注文入力 (P-001)** | 画面上部にカテゴリ切替（スクロール可能なタブ）。<br>中央エリアに商品のグリッド表示。<br>画面下部に「カート (X点) - ¥YYY」のフローティングボタン。タップでカートシートを開く。<br>売切(Sold Out)の商品はグレーアウト＋バッジ表示。 |
| **カート (Sheet)** | 選択中の商品一覧、数量の増減（`+` `-` ボタン）、スワイプ削除。<br>「会計に進む」ボタンで注文確認画面へ。 |
| **注文確認 (P-002)** | 注文内容のリスト、自動適用された割引の表示、手動での割引追加（割引プリセットをシートから選択）。<br>画面下部に「現金」「PayPay」「クレジットカード (Tap to Pay)」の3つの支払ボタンを配置。 |
| **現金会計 (P-003)** | マスタースクリーン。テンキー風またはプリセットの金額ボタンで計算。<br>お釣りが表示され、「会計を完了する」をタップ。 |
| **PayPay決済 (P-004)** | 決済用QRコードを画面いっぱいに表示。画面輝度を一時的に最大化。<br>5秒間隔で決済状態のAPI（ポーリング）を叩き、完了を受信次第自動遷移。 |
| **Tap to Pay (新規)** | ※下記 3.4 参照 |
| **会計完了 (P-005)** | 大きなチェックマークと決済完了のメッセージ、注文内容のサマリー。<br>「次の注文へ」ボタン（または3秒後に自動的に注文入力画面へ戻る）。 |

### 3.3 📜 注文履歴タブ

*   **一覧表示**: 当日の自団体の `Transaction` の一覧を降順（新しい順）で表示。
*   **機能**: 直前の打ち間違いや、お客様からの申し出による「キャンセル」または「返金（PayPay・Stripe等の対応可否要件に準拠）」を行う。
*   *※備考: 複雑な分析はWeb版で行うため、iOS版では「直近の履歴確認とトラブル対応」に特化する。*

### 3.4 💳 Tap to Pay 決済フロー (Stripe Terminal)

1.  注文確認画面で「クレジットカード等 (Tap to Pay)」を選択。
2.  バックエンドの `/transactions` APIを叩き、`paymentMethod: TAP_TO_PAY` (仮) で取引レコード（`status: PENDING`）を作成。
3.  **iOS内で Stripe Terminal SDK を初期化し、NFCリーダー画面を起動**。
    *   画面に「iPhoneの上部にカードをタッチしてください」とAppleネイティブのUIが表示される。
4.  決済が成功すると、Stripeの `PaymentIntent` ID が返却される。
5.  バックエンドの `/transactions/:txnId/complete-tap-to-pay` (新規エンドポイント) を叩き、PaymentIntent ID を送信。
6.  バックエンドで取引ステータスを `COMPLETED` に更新し、iOSアプリは会計完了画面(P-005)へ遷移。

## 4. バックエンドAPI連動仕様

Web版と同じバックエンド（Express + Prisma）のエンドポイントを使用する。
通信には `URLSession` による RESTful API 呼び出しを用いる。（WebSocketによるリアルタイム同期は、POS専用機のため現時点では必須としない。画面表示/更新のタイミングでAPIを再フェッチする。）

### 4.1 必要となる既存APIエンドポイント

*   **Auth**: `POST /auth/login`, `POST /auth/logout`
*   **User/Org**: `GET /users/me`, `GET /organizations`
*   **Products**: `GET /organizations/:orgId/categories`, `GET /organizations/:orgId/products`
*   **Discounts**: `GET /organizations/:orgId/discounts`
*   **Transactions**:
    *   `POST /organizations/:orgId/transactions` (取引作成)
    *   `POST /organizations/:orgId/transactions/:txnId/complete-cash` (現金完了)
    *   `POST /organizations/:orgId/transactions/:txnId/paypay/create` (PayPay完了待ち受け用QR生成)
    *   `GET /organizations/:orgId/transactions/:txnId/paypay/status` (PayPay状態確認ポーリング)
    *   `POST /organizations/:orgId/transactions/:txnId/cancel` (キャンセル・返金)
    *   `GET /organizations/:orgId/transactions` (注文履歴一覧用)

### 4.2 バックエンドへの追加要件 (Task対応)

iOSアプリのTap to Pay実装に伴い、バックエンド側には以下の修正が必要となります。

1.  **Enumの変更**: データベース（または Prisma スキーマ）の `PaymentMethod` Enum に `TAP_TO_PAY` （または `CREDIT_CARD`）を追加。
2.  **新規エンドポイント**:
    *   `POST /api/v1/organizations/:orgId/transactions/:txnId/complete-tap-to-pay`
    *   役割: Stripe Terminal側で決済が完了した後、その事実をPOSシステムに記録し、ステータスを `COMPLETED` に変更する。

## 5. 制約事項・非機能要件

1.  **オフライン対応 (Network Resilience)**
    *   API取得に失敗した場合（電波状況が悪い場合）のために、起動時に取得した「商品マスタ」「カテゴリ」「割引設定」はアプリ内のローカル（SwiftData等）にキャッシュする。
    *   現金決済のみ、オフライン時でもキューに溜め込み、オンライン復帰時にバックグラウンドで一括送信する設計を**第2フェーズ（将来構想）**として検討する。初回リリースでは「オンライン前提」で実装可能とする。
2.  **デバイス要件**
    *   最低サポートOS: **iOS 16.4 以上** (Tap to Pay on iPhone の要件)
    *   NFCチップを搭載した iPhone XS、iPhone SE (第2世代) 以降の端末。
    *   iOSアプリの開発は Swift 5.10 / SwiftUI / Swift Concurrency を使用。
3.  **複数端末の競合**
    *   一つの団体で複数のiOS端末（またはWebブラウザ）を同時にレジとして使用できる。
    *   売り切れ（在庫連動）の厳密な排他制御は行わず、決済APIのレスポンス時にエラー（在庫不足）が返却された場合にユーザーへ通知する楽観的ロックとする。
