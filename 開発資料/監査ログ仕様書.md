# 監査ログ仕様書

**作成日:** 2026年2月8日  
**ステータス:** 草案
**目的:** 不正防止、トラブル時の原因究明、および運営の透明性確保のため、システム内で行われる重要な操作を記録・表示する仕様を定義する。

---

## 1. 監査対象ユーザーと閲覧権限

監査ログは「誰が・いつ・何をしたか」を記録し、適切な権限を持つユーザーが確認できるようにする。

| ロール | 閲覧範囲 | 主な用途 |
| :--- | :--- | :--- |
| **システム管理者** (実行委員会) | **全体** (全団体 + システム全般) | 不正の監視、システム設定変更の追跡、全団体の運営状況確認 |
| **団体管理者** (各店責任者) | **自団体のみ** | スタッフの操作ミス、不正な値引き/在庫調整、返金処理の確認 |

---

## 2. 記録データ形式 (Data Schema)

既存の `AuditLog` テーブルをベースに記録する。

| フィールド名 | 説明 | 備考 |
| :--- | :--- | :--- |
| `id` | ログID | UUID (Primary Key) |
| `userId` | 操作実行者 | Userテーブルへの外部キー |
| `organizationId` | 関連団体ID | 団体に紐づく操作の場合は必須。システム全体の操作はNULL。 |
| `action` | 操作種別 | 定義済みの `AuditAction` コード（後述） |
| `targetId` | 操作対象のID | 操作されたリソース（商品のID、会員のID、取引ID等） |
| `payload` | 詳細情報 | JSON形式。変更前後の値、IPアドレス、ユーザーエージェント等を含む。 |
| `isSystemAdminAction` | 特権操作フラグ | payload内またはDBカラム。システム管理者による操作の場合にtrue。 |

> [!IMPORTANT]
> **isSystemAdminAction の重要性**  
> 実行委員会（システム管理者）による操作を明示的にマークすることで、特定団体への不適切な介入や職権濫用を監視できるようにする。

## 3. 監査対象アクション一覧

### 3.1 認証・ユーザー管理
| アクションコード | 説明 | 閲覧 | 記録内容 (payload) |
| :--- | :--- | :---: | :--- |
| `LOGIN_SUCCESS` | ログイン成功 | システム | IPアドレス, UserAgent |
| `LOGIN_FAILURE` | ログイン失敗 | システム | IPアドレス, 入力されたEmail |
| `ROLE_CHANGE` | メンバーの権限変更 | 団体/システム | 対象ユーザーID, 新旧ロール |
| `MEMBER_JOIN` | 団体への新規加入 | 団体/システム | 招待コード, 加入ユーザーID |
| `MEMBER_LEAVE` | 団体からの脱退 | 団体/システム | ユーザーID, 脱退理由(あれば) |
| `MEMBER_REMOVE` | 管理者による強制脱退 | 団体/システム | 対象ユーザーID, 実行者 |
| `USER_SUSPEND` | アカウント停止/再開 | 団体/システム | 対象ユーザーID, 新旧ステータス, 理由 |
| `PASSWORD_CHANGE` | パスワードの変更 | システム | IPアドレス, 変更日時 |
| `PASSWORD_RESET` | パスワードのリセット | システム | IPアドレス, リセット方式(メール等) |

### 3.2 商品・在庫（不正防止の要）
| アクションコード | 説明 | 閲覧 | 記録内容 (payload) |
| :--- | :--- | :---: | :--- |
| `PRODUCT_CREATE` | 商品の新規作成 | 団体/システム | 商品名, 初期価格, カテゴリ |
| `PRODUCT_UPDATE` | 商品情報(価格名等)変更 | 団体/システム | 変更項目(price, name等), 新旧の値 |
| `PRODUCT_DELETE` | 商品の論理削除 | 団体/システム | 商品名 |
| `STOCK_ADJUST` | 在庫数の手動調整/棚卸 | 団体/システム | 商品名, 新旧在庫数, 調整理由 |

### 3.3 割引設定
| アクションコード | 説明 | 閲覧 | 記録内容 (payload) |
| :--- | :--- | :---: | :--- |
| `DISCOUNT_CREATE` | 割引プリセットの作成 | 団体/システム | 割引名, 種別, 値, 条件 |
| `DISCOUNT_UPDATE` | 割引設定の変更 | 団体/システム | 変更項目, 新旧の値 |
| `DISCOUNT_DELETE` | 割引設定の削除 | 団体/システム | 割引名 |

### 3.4 取引・会計
| アクションコード | 説明 | 閲覧 | 記録内容 (payload) |
| :--- | :--- | :---: | :--- |
| `TRANSACTION_CANCEL` | 取引のキャンセル(完了前) | 団体/システム | 取引ID, 合計金額, カート内容 |
| `TRANSACTION_REFUND` | PayPay返金処理 | 団体/システム | 取引ID, 返金額, 元の注文ID |
| `CASH_REPORT_CREATE` | レジ締め報告 | 団体/システム | 理論売上, 実査売上, 過不足金額 |

### 3.6 権限・ロール管理 (RBAC)
| アクションコード | 説明 | 閲覧 | 記録内容 (payload) |
| :--- | :--- | :---: | :--- |
| `ROLE_CREATE` | カスタムロールの作成 | 団体/システム | ロール名, 初期権限リスト |
| `ROLE_UPDATE` | ロール名の変更等 | 団体/システム | 変更項目, 新旧の値 |
| `ROLE_DELETE` | ロールの削除 | 団体/システム | 削除されたロール名 |
| `ROLE_PERM_SET` | ロールへの権限割り当て | 団体/システム | ロール名, 追加/削除された権限 |
| `PERMISSION_DEF` | 権限マスターの定義変更 | システム | 権限コード, 説明 |

### 3.7 団体・システム管理 (システム管理者専用)
| アクションコード | 説明 | 閲覧 | 記録内容 (payload) |
| :--- | :--- | :---: | :--- |
| `ORG_CREATE` | 新規団体の作成 | システム | 団体名, 初期設定内容 |
| `ORG_INVITE_REGEN` | 招待コードの再発行 | システム | 団体名, 新旧招待コード |
| `SYSTEM_SETTING_MOD` | システム基本設定変更 | システム | 変更項目(イベント日時等), 新旧の値 |
| `SYSTEM_STOP` | システム緊急停止/再開 | システム | 停止フラグの新旧状態 |

## 4. ログの表示・検索要件

### 4.1 団体管理者向け画面 (Organization Admin)
*   サイドバーの「監査ログ」からアクセス。
*   自団体の操作のみを時系列で表示。
*   **フィルタ条件:**
    *   期間 (From - To)
    *   操作者 (スタッフ名)
    *   カテゴリ (商品関連、会計関連、メンバー関連等)
*   **詳細表示:**
    *   JSON形式のペイロードを人間が読みやすい形式（「価格を 300円 -> 350円 に変更」等）に変換して表示。

### 4.2 システム管理者向け画面 (System Admin Portal)
1.  **全体フィード (Global Feed):** 全ての最新ログを表示するが、主にシステム障害や緊急停止(`SYSTEM_STOP`)などの重要イベントの監視に特化させる。
2.  **団体別詳細ビュー (Per-Organization Logs):** 
    *   システム管理者が「各団体の詳細ページ」にアクセスした際、その団体に関連する監査ログのみを表示するタブを設置する。
    *   これにより、特定の店舗で問題が発生した際の調査を効率化する。
3.  **操作者別ビュー (Per-User Logs):**
    *   特定のシステム管理者がどのような操作を（各団体に対して）行ったかを一覧できるビュー。職権濫用の自己点検・外部監査に利用。

## 5. 記録・表示形式の詳細定義 (Payload & Display Design)

各アクションの `payload` 構造と、それを画面上でどのように「人間が読みやすい形式」にレンダリングするかを定義する。

| アクションカテゴリ | Payload 構造 (JSON) | 表示文字列 (テンプレート) |
| :--- | :--- | :--- |
| **ユーザー・権限** | `{ "targetUserId": "...", "oldRole": "STAFF", "newRole": "ADMIN" }` | `[対象ユーザー名] の権限を [oldRole] から [newRole] に変更しました。` |
| | `{ "method": "manual", "ip": "..." }` | `パスワードを変更しました。` |
| **商品管理** | `{ "productId": "...", "name": "たこ焼き", "changes": { "price": { "old": 300, "new": 350 } } }` | `商品「たこ焼き」の価格を 300円 から 350円 に変更しました。` |
| **在庫調整** | `{ "productId": "...", "oldStock": 10, "newStock": 50, "reason": "棚卸し" }` | `商品「たこ焼き」の在庫を手動で 10個 から 50個 に調整しました。理由: 棚卸し` |
| **取引・返金** | `{ "transactionId": "...", "amount": 1000, "reason": "注文ミス" }` | `取引 ID [transactionId] (1,000円) を返金しました。理由: 注文ミス` |
| **RBAC** | `{ "roleName": "店長", "addedPermissions": ["product:delete"], "removedPermissions": [] }` | `ロール「店長」に権限 [product:delete] を追加しました。` |
| **システム管理** | `{ "organizationName": "建築学科3年", "inviteCode": "ABC123" }` | `団体「建築学科3年」を作成しました（招待コード: ABC123）。` |

> [!TIP]
> **フロントエンドでの実装:**  
> i18nライブラリや専用の `AuditLogFormatter` クラスを使用し、`action` コードに応じたテンプレートを選択して `payload` の値を埋め込む形式をとる。

## 6. 追加の改善提案

1.  **機密データのマスク:**
    *   JSONペイロードを保存する際、パスワードリセットトークンや個人情報（住所等があれば、本システムでは限定的だが）は自動的にマスク(`***`)する。
2.  **IPアドレスの記録と異常検知:**
    *   通常と異なるIP（学内ネットワーク外など）からのログインや操作を強調表示する。
3.  **不審な操作の通知:**
    *   短時間での大量の価格変更や返金処理など、事前定義した「異常パターン」に合致する操作が行われた際、管理者に通知を送る仕組みを検討する。

---

## 5. 実装に関する技術指針

1.  **自動記録ミドルウェア/デコレータ:**
    *   コントローラーまたはサービスレイヤーで、DB更新が成功した直後に非同期で `AuditLog` を作成する共通ロジックを実装する。
2.  **不変性の担保:**
    *   監査ログは追加(`INSERT`)のみを許容し、更新(`UPDATE`)や削除(`DELETE`)はアプリケーションレベルで禁止する。
3.  **パフォーマンス:**
    *   ログの書き込みが主処理（会計完了等）のレスポンス時間を悪化させないよう、バックグラウンド処理（非同期）を検討する。
4.  **保存期間:**
    *   学園祭の事後確認期間を考慮し、最低1年間は保持するよう設定する。
