# 団体脱退機能 仕様書（案）

ユーザーが所属している団体から自発的に脱退するための機能仕様です。

## 1. API仕様

### エンドポイント
`DELETE /api/v1/users/me/organizations/:organizationId`

### リクエストパラメータ
- `organizationId` (URLパラメータ): 脱退したい団体のUUID

### レスポンス

**成功時 (200 OK):**
```json
{
  "success": true,
  "data": {
    "message": "脱退しました",
    "organizationId": "脱退した団体ID"
  }
}
```

**エラー時:**

| ステータスコード | エラーコード | 条件/メッセージ |
| :--- | :--- | :--- |
| **404 Not Found** | `ORG_NOT_FOUND` | 指定された団体IDが存在しない、またはユーザーがその団体に所属していない場合。 |
| **400 Bad Request** | `CANNOT_LEAVE_AS_LAST_ADMIN` | ユーザーがその団体の**唯一の管理者（ADMIN）**である場合。組織の管理不在を防ぐため、脱退前に他のメンバーに権限を委譲する必要があります。 |
| **400 Bad Request** | `CANNOT_LEAVE_SYSTEM_ADMIN_ORG` | システム管理者（System Admin）が自らの所属（実行委員会等）から脱退しようとした場合？（※要検討: システム管理者は別フラグで管理されているため、この制限は不要かもしれませんが、安全のため検討） |

## 2. ビジネスロジック・制約

1.  **所属確認**:
    - リクエストを実行したユーザーが、指定された `organizationId` のメンバーであることを確認します。
    
2.  **管理者保護 (Last Admin Protection)**:
    - ユーザーのロールが `ADMIN` である場合、その団体内に**他に有効な `ADMIN` が存在するか**を確認します。
    - もし自分以外に管理者がいない場合、脱退は拒否されます（エラー: `CANNOT_LEAVE_AS_LAST_ADMIN`）。
    - **解決策**: ユーザーは「メンバー管理画面」で他のユーザーを管理者に昇格させてから、再度脱退処理を行う必要があります。

3.  **データ整合性**:
    - `UserOrganization` テーブルから該当レコードを物理削除します。
    - 過去の取引履歴（Transaction）やログ（AuditLog）の `userId` はそのまま残ります（参照整合性が維持される場合）。または、ユーザーIDは残るが紐付け情報は消える状態となります。

## 3. ユーザーインターフェース (UI)

- **配置場所**:
    - 「設定」ページ (`/settings`) の「所属団体管理」セクション（新規追加）。
- **表示内容**:
    - 所属している団体の一覧リスト。
    - 各行に「脱退」ボタン（またはゴミ箱アイコン）。
- **確認フロー**:
    - 「脱退」ボタン押下時に確認モーダルを表示。
    - 「本当に {団体名} から脱退してもよろしいですか？この操作は取り消せません。」
    - 「脱退する」ボタン（赤色/Destructive）で実行。
