### 1.3 セキュリティ要件の厳守

> [!CAUTION]
> **以下のセキュリティ要件は絶対に実装すること**

#### 1.3.1 認証・認可

**MUST:**
- 全てのAPIエンドポイント（`/auth/*` 以外）にJWT認証を適用
- `organizationId` による団体データ分離を実装
- `isSystemAdmin` フラグによるシステム管理画面のアクセス制御

**実装パターン:**
```typescript
// ❌ 悪い例: 認証なし
app.get('/api/v1/products', async (req, res) => { ... });

// ✅ 良い例: 認証ミドルウェア適用
app.get('/api/v1/products', authenticate, async (req, res) => { ... });
```

#### 1.3.2 PayPay Webhook署名検証

**MUST:**
- `POST /api/v1/webhooks/paypay` で `X-PayPay-Signature` ヘッダーを検証
- クライアントからの「決済完了通知」は絶対に信用しない
- Webhookのみを信頼する

**実装パターン:**
```typescript
// ✅ 必須: Webhook署名検証
const isValid = verifyPayPaySignature(req.body, req.headers['x-paypay-signature']);
if (!isValid) {
  return res.status(401).json({ error: 'Invalid signature' });
}
```

#### 1.3.3 パスワード

**MUST:**
- `bcrypt` を使用してハッシュ化
- 平文パスワードは絶対にDB保存しない

### 1.4 命名規則の厳守

> [!IMPORTANT]
> **一貫性のある命名はバグ防止とメンテナンス性向上の鍵**

#### 1.4.1 データベース

| 項目 | 規則 | 例 |
|------|------|-----|
| モデル名 | PascalCase | `User`, `Organization`, `TransactionItem` |
| フィールド名 | camelCase | `userId`, `organizationId`, `paypayOrderId` |
| Enum値 | SCREAMING_SNAKE_CASE | `ADMIN`, `STAFF`, `COMPLETED` |

#### 1.4.2 API

| 項目 | 規則 | 例 |
|------|------|-----|
| エンドポイント | kebab-case | `/api/v1/organizations/:orgId/cash-reports` |
| パラメータ | camelCase | `{ organizationId, totalAmount }` |

#### 1.4.3 React コンポーネント

| 項目 | 規則 | 例 |
|------|------|-----|
| コンポーネント名 | PascalCase | `OrderInputScreen`, `PaymentMethodSelect` |
| ファイル名 | PascalCase.tsx | `OrderInputScreen.tsx` |
| 画面ID参照 | 設計書に従う | `P-001`, `O-003`, `S-001` |

## 3. 禁止事項（MUST NOT）

### 3.1 設計書にない機能の追加

> [!CAUTION]
> **勝手な機能追加は禁止**

**禁止例:**
- ❌ スタッフによる自由な値引き入力（要件で明示的に禁止）
- ❌ 設計書にないAPIエンドポイントの追加
- ❌ DB設計書にないテーブル追加

**対応:** ユーザーに確認し、設計書を更新してから実装

### 3.2 直接DBアクセス

> [!WARNING]
> **必ず Prisma を経由すること**

**禁止例:**
```typescript
// ❌ 悪い例: 生SQL
const result = await db.$queryRaw`SELECT * FROM users WHERE email = ${email}`;
```

**推奨例:**
```typescript
// ✅ 良い例: Prisma
const user = await prisma.user.findUnique({ where: { email } });
```

### 3.3 技術スタック外のライブラリ導入

**禁止例:**
- ❌ `axios` の代わりに `fetch` のみ使用
- ❌ `shadcn/ui` の代わりに Material-UI を導入
- ❌ TailwindCSS以外のCSSフレームワーク（Bootstrap等）

**対応:** 技術スタック.md に記載がない場合、ユーザーに確認

## 4. トランザクション処理の原則

### 4.1 在庫減算の整合性

> [!IMPORTANT]
> **在庫減算は必ずトランザクション内で行う**

**理由:** 同時操作による在庫の不整合を防ぐ

**実装パターン:**
```typescript
// ✅ 良い例: トランザクション使用
await prisma.$transaction(async (tx) => {
  // 1. 在庫確認
  const product = await tx.product.findUnique({ where: { id: productId } });
  if (product.stock < quantity) throw new Error('INSUFFICIENT_STOCK');
  
  // 2. 在庫減算
  await tx.product.update({
    where: { id: productId },
    data: { stock: { decrement: quantity } }
  });
  
  // 3. StockLog記録
  await tx.stockLog.create({
    data: { productId, changeAmount: -quantity, reason: 'SALE' }
  });
  
  // 4. Transaction作成
  await tx.transaction.create({ ... });
});
```

### 4.2 取引キャンセル時の在庫戻し

**MUST:** 
- キャンセル時は在庫を戻す
- `StockLog` に `reason: 'RETURN'` で記録

## 5. Socket.io リアルタイム通信

### 5.1 イベント命名規則

| イベント名 | 発火タイミング | ペイロード |
|-----------|--------------|-----------|
| `join:organization` | 団体ルーム参加 | `{ organizationId }` |
| `transaction:completed` | 取引完了 | `{ transactionId, organizationId }` |
| `product:updated` | 商品情報更新 | `{ productId, organizationId }` |
| `product:soldout` | 売り切れ | `{ productId, organizationId }` |

### 5.2 軽量化の原則

**SHOULD:** 
- 全データを送信せず、IDのみを送信
- クライアント側でAPIを再フェッチ

**理由:** Socket.ioの負荷軽減（要件定義書 6.1参照）

## 14. 実装開始前の確認（CRITICAL）

> [!CAUTION]
> **実装計画承認後も、必ずユーザーの明示的な指示を待つこと**

**ルール:**
- 実装計画（implementation_plan.md）が承認されても、**自動的に実装を開始してはならない**
- ユーザーから「実装を開始してください」「Phase Nを開始してください」等の明示的な指示があるまで待機
- 承認 = 計画内容への同意であり、実装開始の許可ではない

**対応:**
- 計画承認後は、ユーザーの次の指示を待つ
- 質問や追加の確認事項がある場合のみ、先にユーザーに確認
